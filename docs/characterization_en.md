# 3 Characterization

We present the first systematic measurement study of resource usage patterns in production-grade AI coding agents. Our goal is to answer the following research questions: **RQ1**: What is the execution model of an agent? **RQ2**: What are the resource usage characteristics of agent workloads? We further discuss the fundamental differences between AI coding agent workloads and traditional containerized workloads, and how these differences affect the applicability and efficiency of existing resource management mechanisms.

## 3.1 Experimental Setup

**Platform.** All experiments were conducted on a machine with an Intel Core Ultra 9 285K processor (24 cores, up to 5.8 GHz), 128 GB DDR5 memory, Ubuntu 24.04.3 LTS, and Linux kernel 6.15.11 with cgroup v2 enabled. Each task was executed in an isolated Podman container, and container images came from the official SWE-rebench Docker images, with per-image sizes ranging from 2.9 GB to 17.3 GB. During characterization, we applied no resource limits to containers, so that the measurements reflect the true resource demand of unconstrained execution.

**Dataset.** We used a two-level dataset strategy. First, we collected 111 tasks from the SWE-rebench dataset using a local GLM model and 33 tasks using the cloud-based Haiku model for large-scale statistical analysis. In addition, we selected 18 representative tasks spanning six categories (`CLI_Tools`, `DevOps_Build`, `ML_Scientific`, `Medical_Bio`, `SQL_Data`, and `Web_Network`) and three difficulty levels (`Easy`, `Medium`, and `Hard`) for category-level analysis. These tasks cover typical AI coding agent scenarios, including command-line tool repair, build-system configuration, machine learning debugging, biomedical data processing, database query optimization, and web-service repair.

**Agent implementation.** All tasks were executed with the same agent framework, Claude Code, but with two different underlying models: (1) Haiku, where LLM inference is executed through the Anthropic cloud API; and (2) GLM 4.7 flash, where LLM inference runs locally on a GPU. The agent framework itself, including tool invocation logic, sandboxing, and the Node.js runtime, remained identical across both settings. The only difference was the underlying model and where inference happened. We chose these two models to observe how model choice and inference location (remote API vs. local GPU) affect resource usage inside the container.

**Data collection.** For each task execution, we sampled CPU utilization and memory usage for every container at 1-second intervals, and recorded the type, start time, and end time of every tool call. All tasks were executed in the same sandbox environment to ensure comparability.

## 3.2 RQ1: Agent Execution Model

The execution pattern of an agent is fundamentally different from that of traditional containerized workloads. Unlike serverless or FaaS functions, which handle short stateless requests (100ms-2s), each agent task runs for about 10 minutes on average (10.8 minutes for GLM, 5.8 minutes for Haiku, with a median of 8.1 minutes; see Fig-exec (a)) and performs multiple rounds of stateful reasoning and tool-calling loops. In addition, container startup introduces a non-trivial fixed cost, averaging 26.5 seconds (median 23.0 seconds) and reaching up to 97 seconds.

**About 60% of time is spent on inference and 40% on tool execution, but variance across tasks is huge (0%-86%).** Agent execution alternates between LLM inference and tool execution. As shown in Fig-exec (b), across all 144 tasks (33 Haiku + 111 GLM), when normalized by `active_time` (the span from the first to the last trace record, excluding container startup), the median fraction of time spent in tool execution is about 35% (34.7% for Haiku and 36.5% for GLM), while the mean is 42.5% for Haiku and 36.4% for GLM. The similar medians across both models indicate that a roughly 60/40 split between LLM reasoning and tool execution is a common structural characteristic of agent workloads. However, this ratio varies widely across tasks, from 0% to 86% with a median of about 36%, as shown in Fig-tool-time (a).

**In end-to-end latency, LLM thinking accounts for only 26%-44%; the rest is infrastructure overhead.** When end-to-end latency is normalized by `claude_time` (the full container lifetime), container startup accounts for 31%-48% of the total (47.7% for Haiku and 31.0% for GLM), tool execution accounts for about 26% for both models (25.9% for Haiku and 25.5% for GLM), and actual LLM thinking accounts for only 26% in Haiku and 44% in GLM. In other words, infrastructure overhead (container startup plus tool execution) occupies 57%-74% of end-to-end latency. This finding shows that OS-level container and resource management directly affects more than half of the perceived latency of agent tasks. Optimizing resource control during tool execution is therefore not only about efficiency, but also about user-visible completion time.

![Fig-exec: Task execution time distribution (a) and execution-phase breakdown (b)](../../analysis/comparison_figures/exec_overview.pdf)

**Bash and Task account for more than 90% of tool execution time, but model-specific tool strategies differ substantially.** As shown in Fig-tool-type, we first examine tool-call structure for the Haiku agent across 33 tasks (Fig-tool-type (a)). Task calls (sub-agent invocations) and Bash account for 47.8% (n=17) and 43.2% (n=410) of total tool execution time, respectively. WebSearch and WebFetch together account for about 5%, while the remaining tools (Read, Grep, Edit, and others) together contribute less than 5%. In contrast, the GLM agent relies almost entirely on Bash calls, which account for 99.5% of tool time (n=3304), and does not use Task-style sub-agents or web-search tools. Looking further into the semantic categories of Bash commands (Fig-tool-type (b)), test execution (`pytest`, `unittest`, etc.) accounts for 72.9% of total Bash time in Haiku, while package installation accounts for 10.8%. In GLM, the distribution is more dispersed: 43.7% for test execution, 26.9% for Python snippets, and 10.1% for package installation. Different tool types therefore carry very different resource implications: test execution tends to be CPU- and memory-intensive, whereas file exploration and Git operations are relatively lightweight.

![Fig-tool-type: Distribution of tool execution time (a) and semantic breakdown of Bash commands (b), GLM agent](../../analysis/comparison_figures/tool_bash_breakdown.pdf)

**Execution times across tool types span three orders of magnitude.** In Haiku, Task sub-agents have an average duration of 100.47 seconds (n=17), while Bash commands average 3.76 seconds (n=410). In GLM, Bash commands average 5.93 seconds (n=3304). In comparison, Read operations average only 0.34s in Haiku and 0.08s in GLM, while Edit operations average 0.05s in Haiku and 0.04s in GLM. This difference spans roughly three orders of magnitude, indicating that different tool types require different resource provisioning strategies. The difference in tool strategies across models also has implications for resource management: Haiku delegates part of the work to external services through Task sub-agents and web search, while GLM concentrates almost all computation into local Bash execution, resulting in significantly higher local resource consumption (19598 seconds of total Bash time in GLM vs. 1543 seconds in Haiku).

**Tool calls follow a "understand -> modify -> validate" temporal pattern.** As shown in Fig-tool-time (b), when execution is divided into 10 equal-length phases and tool usage is analyzed over progress, Read operations concentrate in the early phase (the first 30%), corresponding to code understanding; Bash calls are most dense in the middle-to-late phase (40%-80%), corresponding to testing and validation; and Edit operations are distributed relatively evenly throughout the entire run. This phase structure aligns well with the standard software engineering workflow of understanding, modifying, and validating code, and provides a basis for phase-aware resource control.

## 3.3 RQ2: Resource Usage Characteristics

Building on the execution model described in RQ1, we now examine the resource-usage characteristics of agent workloads. We first present a high-level picture of resource consumption, and then analyze its unpredictability from three perspectives: temporal dynamics, non-determinism, and heterogeneity.

**Memory, not CPU, is the decisive bottleneck for multi-tenant concurrency density.** Average CPU utilization is low for agent workloads (13.2% for Haiku and 7.6% for GLM, measured against a single-core baseline), and CPU is far from saturated on our 24-core machine. Memory is the real bottleneck for concurrency density: peak memory usage reaches 2-4 GB, which means that a 128 GB node can sustain only 32-64 instances if provisioned at peak demand, while total CPU usage still remains below 36% of machine capacity. This imbalance, where CPU remains largely idle while memory is limiting, implies that dynamic memory management is key to improving concurrency density: memory should be elastically expanded for short-lived spikes and reclaimed during idle periods to support more concurrent instances.

**Disk storage demands are far higher than in traditional containerized applications.** As shown in Fig-resource (a), across the deduplicated set of 114 unique images, Docker image sizes range from 2.9 GB to 17.3 GB, with an average of 4.1 GB and a median of 3.5 GB. The distribution is heavily concentrated in the 3-4 GB range. For the 111 GLM tasks alone, total image footprint reaches 456 GB, creating substantial storage pressure for multi-tenant deployment.

**Resource consumption has a two-layer structure: a ~185 MB framework baseline plus tool-triggered bursts.** As shown in Fig-resource (b), the agent framework itself (for example, the Claude Code Node.js runtime) maintains an irreducible memory baseline. Across all 144 tasks, the average memory in the early phase (the first 10% of samples) is 183 MB for Haiku and 188 MB for GLM, with a median of about 185 MB. Even during pure LLM inference, when no tool is running, memory remains near this baseline. In contrast, the dramatic resource fluctuations come almost entirely from tool-invoked child processes: test execution, dependency installation, and similar operations can drive memory up to 500 MB-2 GB before it falls back to the baseline. Aggregating normalized memory trajectories over execution progress (Fig-resource (b)) shows that the first half of execution (0%-50%) remains close to a stable baseline of about 185-200 MB, while the second half rises and becomes more variable, consistent with the more intensive Bash phase. In a multi-tenant setting, this means that 64 concurrent instances already require about 12 GB of memory for the framework baseline alone, and that the true challenge lies in the superposition of tool-triggered bursts. Resource-control policies therefore need to distinguish between these two layers: stable protection for the framework baseline, and elastic scheduling for tool bursts.

**Resource consumption is determined by tool semantics, not by tool type; even Bash can differ by 13.7x.** The same tool type, such as Bash, can produce radically different resource usage depending on the task. Among the 18 representative tasks, Bash in the `Medical_Bio` category consumes an average of 4 GB peak memory, while Bash in `Web_Network` requires only 291 MB, a 13.7x difference. This shows that demand is not determined by the tool type itself, but by the semantics of what the tool is doing. Test execution, data loading, and model inference may all appear as Bash calls, yet their resource profiles are completely different. This finding has direct implications for resource control: provisioning based only on tool type is insufficient, and finer-grained semantic awareness is necessary. Quantifying resource bursts by Bash command class further shows that test execution (`pytest` and similar) has a P95 memory spike of 518 MB in Haiku and 234 MB in GLM, with an average CPU spike of +3.2%; package installation has a P95 memory spike of about 233 MB; while file exploration and Git operations have average memory spikes of only 4.5 MB and 13.5 MB. Even within Bash, different semantic categories differ by two orders of magnitude in memory burst amplitude (518 MB vs. 4.5 MB), directly motivating differentiated resource quotas based on command semantics.

![Fig-resource: Docker image size distribution (a) and aggregated memory trajectory (b), all 144 tasks](../../analysis/comparison_figures/resource_profile.pdf)

The unpredictability of these resource characteristics comes from three dimensions: temporal dynamics, non-determinism, and heterogeneity.

### Temporal Dynamics

Agent workloads exhibit highly bursty time-varying resource behavior. Memory usage can change by as much as 2.9 GB within a single 1-second sampling interval, and CPU utilization shows sharp swings with peaks above 100%, indicating multi-core usage. Resource consumption follows a burst pattern rather than a steady trend.

![Fig-timeseries-haiku: Resource-usage time series for Haiku agent on pre-commit/pre-commit#2524](../../analysis/haiku_figures/rq1_resource_timeseries.pdf)

![Fig-timeseries-glm: Resource-usage time series for GLM agent on sigmavirus24/github3.py#673](../../analysis/qwen3_figures/rq1_resource_timeseries.pdf)

Fig-timeseries-haiku and Fig-timeseries-glm show two representative task traces. The upper plot corresponds to a Haiku agent run on `pre-commit/pre-commit#2524`, and the lower plot to a GLM agent run on `sigmavirus24/github3.py#673`. Each figure has two subplots: CPU utilization in blue and memory usage in green. Red shaded regions mark active tool-call intervals such as Bash, Read, and Edit. Resource bursts are clearly aligned with tool execution: most CPU and memory spikes occur during tool calls such as running tests or installing dependencies, while the intervals between tool calls, corresponding to LLM reasoning, remain relatively stable. These examples also reflect model-specific resource patterns. Haiku runs have more concentrated tool calls, each often accompanied by sharp multi-core CPU spikes above 175%. GLM runs invoke tools more frequently, but CPU remains low for long stretches (around 10%), with most variation appearing in memory. Although exact traces vary across tasks, this tool-driven burst-silence alternation is a common pattern across both datasets.

**98.5% of memory bursts occur during tool calls, so resource volatility is almost entirely tool-driven.** To quantify this, we label every resource sample as either "during tool execution" or "during LLM reasoning" and count memory burst events above 300 MB. In the Haiku dataset, tool execution accounts for only 50.6% of sampled time, but contains 98.5% of memory bursts. In the GLM dataset, tool execution accounts for 35.9% of time but contains 67.3% of memory bursts. In both cases, the burst concentration factor is about 1.9x, meaning that the probability of a memory burst during tool execution is nearly twice what time share alone would suggest. CPU burst attribution is more diffuse: 55.3% of CPU bursts happen during tool execution in Haiku, but only 30.2% in GLM, where some CPU bursts come from local GPU inference scheduling overhead. This asymmetry matters: memory bursts are highly concentrated in tool execution, while CPU bursts are less so. As a result, memory should be elastically managed at tool-call granularity, while CPU requires broader context-aware control.

As shown in Fig-change-rate, the maximum observed memory change rate reaches 3 GB/s, and the maximum CPU change rate exceeds 50%/s. Significant change events account for 1.7%-3.8% of all samples.

![Fig-change-rate: Distribution of CPU and memory change rates, Haiku dataset](../../analysis/haiku_figures/rq1_change_rate_distribution.pdf)

**Resource bursts last only 1-2 seconds, with a peak-to-average ratio up to 15.4x; static limits are unreasonable regardless of how they are set.** Resource bursts are extremely transient, as illustrated by the sharp spikes in Fig-timeseries-haiku and Fig-timeseries-glm. For one representative `Medical_Bio_Hard` task, peak memory reaches 4060 MB while average memory is only 264 MB, yielding an overprovisioning factor of 15.4x. Crucially, this 4 GB peak lasts only 1-2 seconds before falling back to a 230 MB baseline. This spike pattern implies that if static limits are set by peak demand, 98% of the time the allocated memory is wasted; if they are set by average demand, short bursts will trigger OOM. Traditional reactive resizing is too slow to handle such brief spikes.

**The correlation between CPU and memory varies by task (-0.84 to +0.50), so co-directional change cannot be assumed.** As seen in Fig-timeseries-haiku and Fig-timeseries-glm, the relationship between CPU and memory usage is task-dependent. Across all tasks, the average correlation coefficient is -0.39, ranging from -0.84 to +0.50, which shows that CPU peaks and memory peaks often do not happen at the same time. Some tasks exhibit positive correlation, where tool execution raises both CPU and memory together. Others exhibit negative correlation, where CPU-intensive phases require little memory, and vice versa. This task-dependent coupling is harder to manage than a simple positive or negative relationship: resource-control mechanisms cannot assume that CPU and memory demand always move together, and must instead monitor and coordinate both dimensions independently.

Memory peaks tend to appear in the second half of execution, with mean and median around 65% progress (see Fig-resource (b)), but the spread covers the full execution timeline. This unpredictability further complicates static resource allocation.

### Non-determinism

Unlike traditional containerized workloads, agent workloads are highly non-deterministic. Even when exactly the same task is executed multiple times, both the resource usage pattern and the final result can differ significantly. We executed one task (`DevOps_Build_Hard`) three times and observed runtimes of 402 seconds, 222 seconds, and 259 seconds, a difference of 1.8x. More importantly, the three runs produced entirely different solutions, with different code modifications, different numbers of changed files, and even different implementation strategies. This non-determinism originates from stochastic LLM inference and diverse agent decision paths, making resource-demand prediction based on historical runs fundamentally difficult.

**85%-97% of tasks contain retry loops, which consume 7%-20% of execution time and cause cumulative memory growth.** Retry behavior is intrinsic to agent workloads. In the Haiku dataset, 85% of tasks (28/33) contain retry groups, defined as sequences of three or more consecutive Bash calls. In the GLM dataset, this rises to 97% (108/111). GLM has an average of 3.9 retry groups per task, and the longest observed retry sequence contains 56 consecutive retries. Retry time accounts for 20.5% of total execution time in GLM and 7.4% in Haiku. This repeated cycle of execute tests, observe failure, modify code, and test again is a behavioral signature of agents. More importantly, each retry loop retains previous memory context instead of cleaning it up, leading to gradual memory accumulation. In the most extreme case, retries retained an extra 502 MB of memory. This pattern is rare in traditional containerized workloads, but common in agents, and its resource impact cannot be predicted ahead of time.

**LLM token count does not predict resource consumption.** In the Haiku dataset, the correlation between total output tokens and peak memory is only r=-0.14 (r=+0.02 in GLM), indicating essentially no linear relationship. The number of dialogue turns is moderately correlated with runtime (r=+0.57 for Haiku and r=+0.82 for GLM), but almost unrelated to peak memory (r=+0.02 for Haiku and r=+0.11 for GLM). This means that even if we can predict how many tokens or turns an agent will generate, we still cannot infer its memory demand. Resource consumption is driven primarily by the semantics of tool execution, not by the scale of LLM reasoning itself.

### Heterogeneity

Resource demand varies significantly across tasks and across models. Among the 18 representative tasks, peak memory ranges from 197 MB to 4 GB, with a coefficient of variation (CV) of 147%. Tasks in `ML_Scientific` and `Medical_Bio` have substantially higher memory demand than tasks in `CLI_Tools` or `Web_Network`, even though all tasks are executed inside the same kind of container.

Haiku and GLM also show clear differences in CPU utilization for the same tasks: Haiku averages 13.2%, GLM averages 7.6%, a 1.7x difference. Average runtime also differs significantly, at 352 seconds for Haiku versus 664 seconds for GLM. This shows that resource demand depends not only on the task itself, but also on the choice of underlying model.

**Simply changing the underlying model yields a fundamentally different resource profile.** The difference in CPU utilization reflects a deeper issue: even if the agent framework is identical (Claude Code in both cases), changing only the underlying model can produce a very different resource footprint. Haiku relies on cloud API inference, and the API calls, response parsing, and context management consume more local CPU. GLM performs inference on a local GPU, so CPU load inside the container comes almost entirely from tool execution. In GLM, only 0.5% of samples exceed 50% CPU utilization, whereas in Haiku the figure is 8.2%. Even on the same task, the two models show different CPU distributions, runtimes, and memory traces. This means that resource-management strategies cannot treat "agent workload" as a single static category; they must adapt to model-specific resource characteristics.

# 4 Gap Analysis

The previous two sections established the unique resource characteristics of agent workloads. In this section, we place these characteristics in the context of known cloud workload classes and systematically analyze why existing resource-management approaches, ranging from kernel cgroup interfaces to cluster-level autoscaling, are ineffective for agent workloads.

## 4.1 Quantitative Comparison with Known Cloud Workload Types

To better understand the distinctiveness of agent workloads, we compare the findings from RQ1 and RQ2 with three common classes of cloud workloads.

| Dimension | Serverless/FaaS | Microservices | Batch/HPC | **AI Coding Agent** |
|------|----------------|--------|-----------|---------------------|
| Execution time | 100ms-2s \cite{shahrad2020serverless} | Long-running | Minutes-hours | **5-11 minutes** |
| Statefulness | Stateless | Externalized state | Stateful | **Stateful in-process** |
| Memory peak/avg | ~1.5x | 2-3x \cite{cortez2017resource} | ~1x | **15.4x** |
| Determinism | Deterministic | Mostly deterministic | Deterministic | **1.8x variance for the same task** |
| Resource pattern | Flat | Stable + diurnal cycles | Gradually increasing | **Burst-silence alternation** |
| Termination cost | Just retry | Migratable | Lose progress | **Lose all LLM context** |

**Peak-to-average memory ratio is the most striking distinguishing signal.** Shahrad et al. show that Azure Functions typically allocate 256 MB and exhibit very small runtime memory fluctuation \cite{shahrad2020serverless}. Cortez et al. find that most Azure VM workloads remain relatively stable over time, with peak-to-average memory ratios usually around 2-3x \cite{cortez2017resource}. Google Autopilot experience similarly suggests that, for most Borg jobs, resource limits within about 2x of actual peak are sufficient \cite{rzadca2020autopilot}. In contrast, agent workloads reach peak-to-average memory ratios as high as 15.4x (`Medical_Bio_Hard`: 4060 MB peak vs. 264 MB average), far beyond these workload classes. This extreme ratio comes from a unique two-layer structure: a stable framework baseline of about 185 MB during LLM reasoning, plus short 1-2 second memory spikes caused by tool execution, especially test runs and dependency installation.

**Non-determinism breaks history-based prediction.** Traditional cloud workloads are largely predictable. Historical utilization is effective for resource planning in Borg \cite{verma2015borg, tirmazi2020borg}, and Autopilot uses historical P95 measurements successfully for recommendation \cite{rzadca2020autopilot}. Agent workloads violate this assumption because their resource usage is driven by stochastic LLM inference. The same task can be run three times and produce completely different code-modification paths and resource curves, with runtime variance reaching 1.8x. In addition, output token count is almost uncorrelated with peak memory (r=-0.14), and the number of dialogue turns is also unrelated to peak memory (r=+0.02). This means that historical metrics, whether based on moving averages, exponential smoothing, or percentile methods, are fundamentally unreliable for agent workloads.

**Statefulness plus long runtime makes termination extremely costly.** Serverless functions are stateless, so an OOM kill can be handled by simply retrying from scratch, often with only 100ms-2s delay \cite{shahrad2020serverless}. Microservices can often degrade gracefully by externalizing state or rescheduling pods. In contrast, an agent keeps its entire execution context in process memory, including LLM conversation history, intermediate code changes, and tool-call results. One OOM kill means losing about 10 minutes of stateful computation on average, and because of non-determinism, rerunning does not guarantee reaching the same solution path. As a result, any resource-management strategy that relies on "kill and restart", including Kubernetes eviction and systemd-oomd, is disproportionately costly for agent workloads.

## 4.2 Why Existing Resource Management Solutions Fall Short

We now examine five classes of existing resource-management mechanisms, from kernel interfaces to cluster orchestration, and explain the design assumptions behind them and why those assumptions do not hold for agent workloads.

**(1) cgroup v2 hard limits (`memory.max` / `--memory`).** This is the underlying mechanism behind Docker and Kubernetes memory limits: when memory usage reaches `memory.max`, the kernel triggers synchronous OOM kill in the allocation path \cite{cgroupv2}. For traditional workloads with peak-to-average ratios of 2-3x, provisioning to peak causes 50%-67% overprovisioning, which is often acceptable. For agent workloads, with a ratio of 15.4x, peak-based provisioning means that 98% of the time memory stays below 264 MB while 4060 MB is reserved, wasting 93% of the memory. Average-based provisioning, on the other hand, causes 1-2 second tool bursts to trigger OOM and destroy all agent state. Aggregated analysis over all 144 tasks confirms this dilemma, with CPU peak-based overprovisioning factors also reaching 11.1x-13.9x. The core issue is that hard limits express a one-dimensional static threshold and cannot capture the two-layer "low baseline + high burst" structure of agent workloads.

**(2) cgroup v2 soft limits (`memory.high`).** `memory.high` is more flexible: when usage exceeds the threshold, the kernel triggers direct reclaim and throttling rather than OOM \cite{cgroupv2}. This works reasonably well for traditional workloads, where reclaim can recover file cache and throttling can slow allocation without breaking correctness. However, agent workloads break `memory.high` in two structural ways. First, reclaim cannot distinguish framework memory (Node.js heap and V8 JIT caches, about 185 MB, not reclaimable in practice) from child-process memory used by tools such as `pytest` or dependency installers, which can be limited but should not simply be reclaimed indiscriminately. Reclaiming framework pages increases V8 GC pressure and invalidates JIT caches, which slows LLM response parsing and lengthens total runtime. Second, `memory.high` can only be set once per container, so it cannot differentiate between tool calls with radically different needs. `git status` might produce an average spike of 13.5 MB, whereas `pytest` can reach a P95 spike of 518 MB, yet both share the same cgroup.

**(3) PSI-driven user-space mechanisms (systemd-oomd, Meta oomd).** These systems monitor Pressure Stall Information (PSI) metrics and react when memory or I/O pressure exceeds a threshold, usually by killing the cgroup or triggering swap \cite{systemd-oomd, psi}. Their design assumes that (a) there is enough time to act after pressure appears, and (b) kill or migration is an acceptable degradation mechanism. Neither assumption holds for agents. For (a), memory bursts last only 1-2 seconds and can change at 3 GB/s (RQ2), while the full path from PSI generation to user-space daemon wakeup, decision, and cgroup-control-file update already takes tens of milliseconds, by which time the burst may already be over or the kernel may already have intervened. For (b), killing a cgroup is acceptable for stateless services that can automatically restart, but for agents it means losing all accumulated context. More fundamentally, PSI is aggregated at container granularity, so it cannot attribute pressure to a specific tool call; it can only tell the controller that "the container is under pressure", not that "the running `pytest` process is causing the pressure".

**(4) Cluster-level autoscaling (Kubernetes VPA / Autopilot).** Kubernetes Vertical Pod Autoscaler (VPA) recommends resource requests and limits based on historical metric percentiles, and Google Autopilot applies similar ideas in Borg \cite{k8s-vpa, rzadca2020autopilot}. The core assumption is that resource demand is predictable across repeated pod executions, meaning that historical P95 is a reasonable upper bound for the future. Agent workloads violate this in two ways. First, **the same agent shows task-dependent demand**: `CLI_Tools` tasks may peak at 197 MB, while `Medical_Bio` tasks reach 4060 MB, a 20x gap. Any single recommendation is therefore either too high or too low. Second, **even the same task changes across runs**: non-determinism makes historical percentiles statistically unstable. In addition, VPA operates at pod-restart granularity in stable mode, or minute-level in-place resize in alpha mode \cite{k8s-inplace-resize}, while agent bursts occur at second-level timescales. VPA cannot adjust resources dynamically within a single execution.

**(5) Kubernetes QoS classes and CPU bandwidth control.** Kubernetes classifies pods as Guaranteed (`request=limit`), Burstable (`request<limit`), and BestEffort (no request) \cite{k8s-qos}. Guaranteed is equivalent to static provisioning, which we have already shown wastes 91%-93% of resources. BestEffort is the first class to be evicted under node pressure and is therefore unsuitable for stateful agents. Burstable allows elastic use between request and limit and is the closest option to agent needs, but it is still defined as a single interval at container granularity. It cannot distinguish between the near-zero CPU, low-memory LLM reasoning phase and the multi-core, high-memory tool-execution phase, nor can it set differentiated limits for different tool calls inside one container. CFS bandwidth control (`cpu.max`) has the same problem: a flat CPU quota wastes resources during API waiting and throttles execution during tests. Agent CPU demand follows an extreme bimodal distribution. In GLM, only 0.5% of samples exceed 50% CPU utilization, but peak CPU during tool execution exceeds 175%. A single bandwidth quota cannot fit both modes.

## 4.3 Summary and Design Requirements

The above analysis reveals one common limitation across all existing mechanisms: **their control granularity does not match the resource granularity of agent workloads**. Existing systems set a single policy at the container or pod level, while agent resource demand changes at tool-call granularity. Based on RQ1 (execution model), RQ2 (resource characteristics), and the gap analysis in Sections 4.1 and 4.2, we summarize three core characteristics of agent workloads and the resulting design requirements:

**Characteristic 1: a two-layer structure of framework baseline plus tool bursts.** Agents maintain an irreducible framework baseline of about 185 MB, while tool calls add 1-2 second memory spikes up to 15.4x above the average. -> **Requirement R1**: control logic must execute in the kernel memory-allocation path, so that admission or throttling decisions can be made immediately when tool child processes allocate memory, rather than asynchronously in user space after periodic polling.

**Characteristic 2: resource demand changes at tool-call granularity.** Tool execution time spans three orders of magnitude (0.05s-100s), semantic categories span two orders of magnitude in memory spikes (4.5 MB-518 MB), and the temporal pattern follows the software engineering phases of understand, modify, and validate. -> **Requirement R2**: the resource-control domain must align with tool-call boundaries, so that each tool call, or at least each tool category, can have its own policy instead of sharing one container-level limit.

**Characteristic 3: unpredictability from temporal dynamics, non-determinism, and heterogeneity.** Resource change rates reach 3 GB/s, runtime variation for the same task reaches 1.8x, and peak memory differs by 20x across categories. -> **Requirement R3**: policies must support dynamic runtime adaptation based on live observations rather than historical prediction, and they must adapt to model-specific and task-specific resource profiles.

Together, these three requirements, kernel-path execution (R1), tool-call granularity (R2), and runtime adaptivity (R3), define a design space that existing systems do not cover. AgentCgroup is designed specifically for this space, and we describe that system in the next section.
